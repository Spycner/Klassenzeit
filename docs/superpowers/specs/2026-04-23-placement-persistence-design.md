# Placement persistence: `scheduled_lessons` + upsert on POST + GET `/schedule`

**Date:** 2026-04-23
**Status:** Design approved, plan pending.

## Problem

PR #119 shipped `POST /api/classes/{class_id}/schedule`, but each solve is ephemeral. Two consecutive POSTs against the same class produce independent results and the second has no memory of the first. The frontend cannot render a durable schedule, because the backend does not hold one; there is no `GET /schedule` endpoint to read back from. Sprint step 1 in `docs/superpowers/OPEN_THINGS.md` names this explicitly: without placement persistence, the schedule view (sprint step 2) has nothing to render and the Grundschule seed + Playwright smoke (sprint steps 3-4) cannot assert on a saved timetable.

The brainstorm (`/tmp/kz-brainstorm/brainstorm.md`, Q2) ruled out extending `lessons` with nullable `time_block_id` / `room_id` columns: a lesson with `hours_per_week > 1` needs one placement per weekly hour, which is a relation, not a scalar.

## Goal

One backend PR that:

1. Adds a `scheduled_lessons` join table.
2. Makes `POST /api/classes/{class_id}/schedule` persist the per-class placements inside one transaction (replace-all, then insert) without changing the existing response shape.
3. Adds `GET /api/classes/{class_id}/schedule` returning `ScheduleReadResponse { placements: list[PlacementResponse] }`.

Frontend remains on the read path of the existing POST response until the schedule view ships in the next PR.

## Non-goals

- **Frontend schedule view.** Sprint step 2, separate PR.
- **`dayShortKey(n)` helper extraction.** OPEN_THINGS entry under "Pay down alongside the sprint" points to the schedule-view PR as its first caller; no caller today.
- **Persisting violations.** Violations are per-solve diagnostics; a stale violation would mislead. POST still returns them; GET does not.
- **Whole-school persistence in one POST.** Every POST is scoped to one class. Other classes' placements are not mutated, even though the solve sees them for routing.
- **Solver consumption of persisted sibling placements.** Until this lands (follow-up), class-level solves can produce placements that overlap persisted sibling classes' rooms. Documented, not fixed.
- **Advisory locks on concurrent POSTs.** Current concurrency guard is Postgres row-level locks inside the delete-then-insert transaction. Explicit advisory locks deferred.
- **`schedule_runs` history table.** Every POST replaces; there is no audit view to back.
- **`DELETE /api/classes/{class_id}/schedule`.** No UX calls for "wipe schedule" today.

## Design

### Data model

New table `scheduled_lessons`:

```python
class ScheduledLesson(Base):
    """One persisted placement of a lesson-hour in a (time_block, room) slot."""

    __tablename__ = "scheduled_lessons"

    lesson_id: Mapped[uuid.UUID] = mapped_column(
        ForeignKey("lessons.id", ondelete="CASCADE"), primary_key=True
    )
    time_block_id: Mapped[uuid.UUID] = mapped_column(
        ForeignKey("time_blocks.id", ondelete="CASCADE"), primary_key=True
    )
    room_id: Mapped[uuid.UUID] = mapped_column(
        ForeignKey("rooms.id", ondelete="CASCADE")
    )
    created_at: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now()
    )
```

Composite PK `(lesson_id, time_block_id)` matches the join-table convention used by `teacher_qualifications`, `room_availabilities`, `room_subject_suitabilities`, and `teacher_availabilities`. `room_id` is a dependent attribute of the (lesson, block) pairing and therefore not part of the key.

`ON DELETE CASCADE` on all three FKs means deleting a lesson, time block, or room automatically removes any dependent placements. The next POST rebuilds whatever placements are still consistent with the updated schema.

No cross-class uniqueness constraint on `(time_block_id, room_id)` or on `(school_class_id, time_block_id)`. The rationale (brainstorm Q4) is that the solver does not yet feed persisted sibling placements back into the problem, so adding such a constraint would hard-fail the second class's solve as soon as the first class claimed a given (block, room) pair. That trade-off is recorded as a follow-up in OPEN_THINGS.

No denormalised `school_class_id` column. The GET query joins through `Lesson` to scope by class; at prototype size the join is trivial.

### Persist / read helpers

Two new async functions in `scheduling/solver_io.py`:

```python
async def persist_solution_for_class(
    db: AsyncSession,
    class_id: UUID,
    filtered: dict,
) -> None:
    """Replace this class's persisted placements with the filtered solver output.

    Deletes every ``scheduled_lessons`` row whose ``lesson_id`` belongs to the
    class, then inserts one row per placement in ``filtered["placements"]``.
    Runs inside the caller's transaction (the route handler's session).
    """


async def read_schedule_for_class(
    db: AsyncSession,
    class_id: UUID,
) -> list[PlacementResponse]:
    """Return the class's persisted placements, 404 if the class is missing.

    Joins ``scheduled_lessons`` to ``lessons`` on ``lesson_id`` and filters by
    ``school_class_id``. Result order is stable but arbitrary.
    """
```

Both sit next to the existing `build_problem_json`, `run_solve`, `filter_solution_for_class` trio, keeping the solver-IO boundary cohesive. Current file is ~284 lines; adding ~60 keeps it comfortably below the ~400-line informal split threshold.

`persist_solution_for_class` runs inside the ambient session. The route handler calls it before returning, and FastAPI's session dependency commits on successful exit. No explicit `await db.commit()` inside the helper.

### Route wiring

`routes/schedule.py` ends up with two handlers:

```
POST /api/classes/{class_id}/schedule
  (existing) build problem, solve, filter, persist, return ScheduleResponse

GET /api/classes/{class_id}/schedule
  (new) load class (404 if missing), return ScheduleReadResponse with persisted placements
```

The POST handler adds one line after the existing filter step:

```python
filtered = solver_io.filter_solution_for_class(solution, class_lesson_ids)
await solver_io.persist_solution_for_class(db, class_id, filtered)
# ...
return ScheduleResponse.model_validate(filtered)
```

The GET handler is small:

```python
@router.get("/classes/{class_id}/schedule")
async def read_schedule_for_class_route(
    class_id: uuid.UUID,
    _admin: Annotated[User, Depends(require_admin)],
    db: Annotated[AsyncSession, Depends(get_session)],
) -> ScheduleReadResponse:
    """Return the persisted placements for this class (empty list if none)."""
    placements = await solver_io.read_schedule_for_class(db, class_id)
    return ScheduleReadResponse(placements=placements)
```

The 404 "class not found" path is handled inside `read_schedule_for_class` to keep parity with the POST handler's 404 in `build_problem_json`.

### Response schemas

```python
class PlacementResponse(BaseModel):            # unchanged
    lesson_id: UUID
    time_block_id: UUID
    room_id: UUID

class ScheduleReadResponse(BaseModel):         # new
    placements: list[PlacementResponse]

class ScheduleResponse(BaseModel):             # unchanged
    placements: list[PlacementResponse]
    violations: list[ViolationResponse]
```

`ScheduleReadResponse` deliberately omits `violations`. Returning `violations: []` from GET would misrepresent the absence of storage (brainstorm Q7). Frontend readers use the same `PlacementResponse` for both endpoints so the render path is shared.

### Transaction and concurrency

`persist_solution_for_class` issues `DELETE FROM scheduled_lessons WHERE lesson_id IN (SELECT id FROM lessons WHERE school_class_id = :class_id)`, then `INSERT` one row per placement. Both statements execute in the route handler's existing async transaction. Atomicity guarantees no window with half a schedule.

Concurrency behaviour (brainstorm Q10): a second POST for the same class blocks on row-level locks during DELETE, then runs after the first commit. Last writer wins; no explicit `SELECT FOR UPDATE` or advisory lock. Deferred as a follow-up; prototype has a single admin user.

### Control flow

```
POST /classes/{id}/schedule
  ├─ build_problem_json (existing)
  ├─ run_solve          (existing)
  ├─ filter_solution_for_class (existing)
  ├─ persist_solution_for_class (NEW)   ── DELETE + INSERT in same tx
  └─ return ScheduleResponse (existing)

GET /classes/{id}/schedule
  └─ read_schedule_for_class (NEW)      ── SELECT JOIN lessons
      └─ return ScheduleReadResponse
```

### Structured logging

Add two new events to bracket the persist step, matching the existing `solver.solve.*` style:

```python
logger.info("schedule.persist.start", extra={
    "school_class_id": str(class_id),
    "placements_to_persist": len(filtered["placements"]),
})
# ... DELETE + INSERT ...
logger.info("schedule.persist.done", extra={
    "school_class_id": str(class_id),
    "rows_deleted": deleted_count,
    "rows_inserted": inserted_count,
})
```

Keeps observability parity with `solver.solve.start` / `solver.solve.done`.

## Error modes

| Condition | HTTP | Detail |
|---|---|---|
| POST, class not found | 404 | (existing) |
| POST, pre-solve validation | 422 | (existing) |
| POST, empty filtered result | 200 | persists empty set (deletes existing rows) |
| POST, DB write fails | 500 | transaction rolls back; no partial persistence |
| GET, class not found | 404 | `Class not found` |
| GET, no schedule yet | 200 | `{ "placements": [] }` |

Explicitly not surfacing: "stale schedule" (GET returns whatever is persisted without freshness metadata). A `last_solved_at` column is deferred to a later UX pass.

## Testing plan

- `persist_solution_for_class` on an empty class writes the correct rows.
- `persist_solution_for_class` called twice replaces rather than duplicates.
- `persist_solution_for_class` on an empty filtered result clears the class's rows.
- `persist_solution_for_class` for class A does not touch class B's rows.
- `read_schedule_for_class` returns `[]` for a class with no placements.
- `read_schedule_for_class` returns only rows scoped to the requested class when siblings have rows.
- `read_schedule_for_class` returns 404 (via raised `HTTPException`) when the class does not exist.

Extend `backend/tests/scheduling/test_schedule_route.py`:

- Existing POST tests still pass unchanged.
- New: POST then GET returns the same placements.
- New: POST twice, GET after the second call matches the second solve.
- New: POST for class A does not mutate class B's persisted placements.
- New: GET for a never-solved class returns 200 with empty list.
- New: GET for a non-existent class returns 404.

Fixtures: reuse `backend/tests/scheduling/conftest.py` factories (`create_subject`, `create_week_scheme`, `create_time_block`, `create_room`, `create_teacher`, `create_school_class`). ## Risks

- **Cross-class drift.** Persisting per-class means two classes can end up with placements that share the same (time_block, room). The solver doesn't know about persisted siblings yet. Tracked in OPEN_THINGS as "whole-school cross-class consistency".
- **Replace-all semantics.** Every POST wipes the class's existing rows before inserting new ones. If a manual-edit UX ever lands, that UX must write through a different path, not through POST. Safe today because no manual-edit path exists.
- **Stale references.** Cascade deletes on `time_block_id` / `room_id` silently drop placements when the schedule owner reshapes the week scheme. Users see a shrunken schedule until the next POST rebuilds. Acceptable for the prototype.
- **GET response size.** A class with 25 hours/week produces 25 rows. No pagination needed.

## Follow-ups (to add to OPEN_THINGS)

- Feed persisted sibling placements into the solver's `Problem` input so per-class solves respect the school-wide schedule.
- Decide whether to persist solver violations (or return "last-solve violations" on GET) once the schedule view's diagnostics UX is designed.
- Postgres advisory lock around POST for the same class, if demo traffic shows interleaving.
- `DELETE /api/classes/{class_id}/schedule` to wipe persisted placements without re-solving, if a UX need arises.
